Sub-project 1: Infrastructure + UI Rebuild

Choose the architecture approach for the MUI 3.x migration

A

MUI + Emotion (Official Stack) — Recommended

Install @mui/material with Emotion CSS-in-JS. Full Material Design 3 component library with built-in theming, Dynamic Color via @material/material-color-utilities, light/dark mode toggle, and responsive breakpoints.

Pros

  • Largest React component library, well-documented
  • Built-in theme system with Dynamic Color support
  • Active community, frequent updates
  • i18n-friendly — all text is customizable

Cons

  • Bundle size ~80KB gzipped for core
  • Emotion runtime CSS-in-JS adds overhead
  • MUI v6 still transitioning to full MD3 parity
B

MUI + Pigment CSS (Zero-Runtime)

Use MUI with the newer Pigment CSS engine — zero-runtime CSS extraction at build time. Better performance but newer and less battle-tested.

Pros

  • No runtime CSS overhead — faster page loads
  • Same MUI components and API
  • Better for SSR/SSG if needed later

Cons

  • Still experimental in MUI v6
  • Dynamic theming (runtime color changes) harder
  • Less community support for issues
C

Material Web Components + Custom React Wrappers

Use Google's official @material/web Web Components (true MD3) with thin React wrappers. Closest to the MD3 spec but requires more custom work.

Pros

  • True Material Design 3 — Google's own implementation
  • Framework-agnostic, future-proof
  • Smaller bundle per component

Cons

  • Web Components + React integration has friction
  • Smaller ecosystem, fewer ready-made components
  • Must build many patterns from scratch (data tables, complex forms)
